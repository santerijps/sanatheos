# Sanatheos ‚Äî Project Context

Everything needed to understand and rebuild this project from scratch.

## Overview

Sanatheos is a fast, offline-capable Bible reader and search application built with TypeScript and Bun. The name combines *sana* (Finnish: "word") and *theos* (Greek: "God"). It runs as a single-page application with no framework dependencies ‚Äî just vanilla TypeScript compiled to a single browser bundle. It can be served from a Bun HTTP server during development or deployed as a static site (GitHub Pages).

## Bible Data Format

Each translation is stored as a single JSON file in `public/text/` (e.g., `NHEB.json`, `KJV.json`). The source format is an array-based structure:

```json
{
  "translation": "NHEB",
  "books": [
    {
      "name": "Genesis",
      "chapters": [
        {
          "chapter": 1,
          "verses": [
            { "verse": 1, "text": "In the beginning, God created the heavens and the earth." },
            { "verse": 2, "text": "The earth was formless and empty. ..." }
          ]
        }
      ]
    }
  ]
}
```

The `loadBible()` function in `bible-loader.ts` converts this array-based format into the internal `BibleData` object format:

```json
{
  "Genesis": {
    "1": {
      "1": "In the beginning, God created the heavens and the earth.",
      "2": "The earth was formless and empty. ..."
    },
    "2": { ... }
  }
}
```

Keys are: `{ [bookName: string]: { [chapter: string]: { [verse: string]: string } } }`.

Book names are normalized via `SOURCE_NAME_MAP` (e.g., "I Samuel" ‚Üí "1 Samuel", "Psalms" ‚Üí "Psalm"). Empty verses are trimmed and empty books are skipped. Books are ordered according to `BOOK_ORDER` (OT ‚Üí Deuterocanonical ‚Üí NT). A `translations.json` manifest lists available translation codes as a JSON array.

The build script generates per-translation `bible-CODE.json` files for the `docs/` output.

## Description Data Format

Description files live in `public/data/` named by language: `descriptions-en.json`, `descriptions-fi.json`.

```json
[
  {
    "name": "Genesis",
    "description": "This book is so called from its treating of the generation...",
    "chapters": [
      { "number": 1, "description": "God creates Heaven and Earth, and all things therein, in six days." },
      { "number": 2, "description": "God rests on the seventh day and blesses it..." }
    ]
  }
]
```

Book names use English keys (matching Bible data keys). Descriptions are displayed in the UI when viewing full chapters or books. The build script copies the entire `data/` directory (which includes descriptions) to `docs/data/`. The server serves them at `/data/descriptions-LANG.json` (e.g., `/data/descriptions-en.json`). The `fetchDescriptions(code)` function in `app.ts` maps translation codes to language codes via `TRANSLATION_LANG` (NHEB‚Üí"en", KJV‚Üí"en", CPDV‚Üí"en", KR38‚Üí"fi").

**Psalm numbering:** The descriptions originate from the Catholic Public Domain Version (CPDV), which uses Vulgate/Septuagint psalm numbering. Since WEB and KR38 use Hebrew psalm numbering, the Psalm chapter descriptions have been remapped from Vulgate to Hebrew numbering. Key differences: Vulgate Psalms 9‚Äì10 merge into Hebrew 9 (Hebrew 10 left empty), Vulgate 114‚Äì115 merge into Hebrew 114 (Hebrew 115 empty), Hebrew 116 combines Vulgate 114+115, Hebrew 117‚Äì146 shift by +1 from Vulgate 116‚Äì145, Hebrew 147 combines Vulgate 146+147, and Hebrew 148‚Äì150 match Vulgate.

## Stories Data Format

The Bible stories list lives in `public/data/stories.json`. It is fetched lazily (only when the Stories pane is first opened) and cached in memory. Structure:

```json
[
  {
    "id": "creation",
    "title": "The Creation",
    "title_fi": "Luominen",
    "description": "God creates the heavens, earth, light... in six days.",
    "description_fi": "Jumala luo taivaan, maan, valon...",
    "ref": "Genesis 1-2",
    "category": "Old Testament"
  }
]
```

Fields:
- `id` ‚Äî unique slug identifier
- `title` / `title_fi` ‚Äî story title in English and Finnish
- `description` / `description_fi` ‚Äî one-sentence summary in both languages
- `ref` ‚Äî Bible reference string passed directly to the search input on click (e.g., `Genesis 1-2`, `Matthew 5-7`)
- `category` ‚Äî one of `"Old Testament"`, `"New Testament"`, `"Deuterocanonical"`

Language selection follows the active UI language. The build script copies the entire `data/` directory (including `stories.json`) to `docs/data/`.

## Styleguide Data Format

The `styleguide.json` file (in `public/` and `docs/`) contains paragraph and poetry formatting metadata derived from USFM (Unified Standard Format Markers) source files of the World English Bible (WEB) translation. It maps books ‚Üí chapters ‚Üí formatting instructions:

```json
{
  "Genesis": {
    "1": {
      "paragraphs": [1, 3, 6, 9],
      "poetry": {},
      "stanzaBreaks": []
    }
  },
  "Psalms": {
    "1": {
      "paragraphs": [],
      "poetry": { "1": 1, "2": 2, "3": 1 },
      "stanzaBreaks": [5]
    }
  }
}
```

- `paragraphs` ‚Äî verse numbers where a new paragraph begins (from USFM `\p` markers).
- `poetry` ‚Äî verse numbers mapped to indentation level 1‚Äì3 (from USFM `\q1`, `\q2`, `\q3` markers).
- `stanzaBreaks` ‚Äî verse numbers preceded by an empty line / stanza break (from USFM `\b` markers).

The styleguide is applied across all translations (not just WEB), providing consistent visual formatting. The file is pre-generated and committed ‚Äî the original USFM source files are not part of the repository.

## Subheadings Data Format

Subheading files live in `public/data/` named by language: `subheadings-en.json`, `subheadings-fi.json`. Structure:

```json
{
  "Genesis": {
    "1": [{ "v": 1, "t": "The Creation" }, { "v": 6, "t": "The Second Day" }],
    "2": [{ "v": 1, "t": "The Seventh Day" }]
  }
}
```

Type: `SubheadingsData = Record<string, Record<string, SubheadingEntry[]>>` where `SubheadingEntry = { v: number; t: string }`. `v` is the verse number where the subheading appears, `t` is the subheading text.

Subheadings are loaded on app startup via `fetch("./data/subheadings-LANG.json")` and stored in `render.ts` via `setSubheadings()`. In parallel mode, secondary translation subheadings are loaded separately and stored via `setSecondarySubheadings()`, so each column displays subheadings in its own language.

English and Finnish files have matching structure (same book keys, same number of entries per book). There are approximately 1400 entries per language covering all 66 books.

Subheadings are rendered by `renderStyledVerses()` as `<div class="subheading">` elements above the relevant verse.

## Module-by-Module Description

### app.ts ‚Äî Main Entry Point

The `init()` function runs on page load and orchestrates everything:

1. **Reads initial state** from URL query parameters via `readState()`.
2. **Loads Bible data** ‚Äî checks IndexedDB cache first, then fetches `bible-CODE.json` from the server. Stores in IndexedDB for offline use.
3. **Loads metadata in parallel** ‚Äî fetches `data/styleguide.json`, `data/descriptions-LANG.json`, and `data/subheadings-LANG.json` concurrently via `Promise.all()` and passes them to their respective setters in render.ts. For parallel translations, also loads secondary subheadings via `setSecondarySubheadings()`.
4. **Initializes the search engine** with the loaded data.
5. **Populates settings selectors** ‚Äî translation, parallel, language, theme, font size ‚Äî from `localStorage` and URL state.
6. **Renders initial content** based on URL state.
8. **Wires up all event listeners:**
   - Search input with 150ms debounce and auto-closing double quotes.
   - Index panel open/close with scroll locking.
   - Unified side panel (`#side-overlay`) open/close/tab-switching with scroll locking.
   - Keyboard shortcuts (Escape, Ctrl+K, Ctrl+B, Ctrl+I).
   - Content click handlers (nav arrows, search results, chapter links, headings, copy buttons).
   - Verse context menu (left-click on verse numbers, long-press on touch).
   - Swipe navigation on touch devices.
   - Browser back/forward (`popstate`).
   - Translation switching with automatic query book name translation.
   - Parallel translation loading and toggling.
   - Theme and font size changes.

**Unified side panel** ‚Äî A single `#panel-btn` in the header opens `#side-overlay`, which contains `#side-panel` (a flexbox row). The panel has:
- `#side-tab-rail` ‚Äî 52px icon column with three `.side-tab-btn[data-tab]` buttons (stories, settings, info) and `#side-close`
- `#side-content` ‚Äî scrollable area with three `.side-pane[data-pane]` divs; only the `.active` pane is visible (`display: flex`)

Key functions:
- `activateSideTab(tab)` ‚Äî toggles `.active` on both tab buttons and panes, saves to `localStorage` key `"side-panel-tab"`
- `openSidePanel(tab?)` ‚Äî adds `.open` to `#side-overlay`, calls `activateSideTab`, loads stories data if opening stories tab
- `closeSidePanel()` ‚Äî removes `.open` from `#side-overlay`

**Stories data flow** ‚Äî `loadStoriesData()` fetches `./data/stories.json` once and caches the result in a module-scoped variable. `renderStoriesList(stories, filter)` builds HTML from the filtered list, grouping entries by `category` with `.stories-category-label` headers and `.story-item` buttons. Clicking a story calls `closeSidePanel()`, sets the search input value to the story's `ref`, and dispatches an `input` event to trigger navigation.

**Key design choice:** The app never routes to different pages. All navigation updates `#content` innerHTML and pushes URL state. The URL uses query parameters (`?t=NHEB&book=gen&chapter=3&verse=16`) not hash fragments.

**Translation switching logic:** When changing translations, the app calls `parseQueryBooks()` *before* switching to capture English book keys from the old translation's aliases, then rebuilds the query string using the new translation's display names. This ensures `"Joh 3:16"` becomes `"John 3:16"` when switching KR38‚ÜíNHEB.

**State flow:** User input ‚Üí `applyState()` ‚Üí rendering function ‚Üí `replaceState()`/`pushState()` ‚Üí URL updated. On `popstate`, the reverse: URL ‚Üí `readState()` ‚Üí `applyState()`.

**`applyState()` dispatch:** If the query parses as pure navigation references (via `tryParseNav`) and all books exist in data, it navigates directly. Otherwise it calls `search()` and renders results ‚Äî this handles both quoted text searches and unquoted reference-style queries that `tryParseNav` rejected.

**Translation metadata** is hardcoded in `TRANSLATION_NAMES` and `TRANSLATION_LANG` constants:
- `NHEB` ‚Üí English (default), `KJV` ‚Üí English, `CPDV` ‚Üí English, `KR38` ‚Üí Finnish (Suomi)

### search.ts ‚Äî Search Engine

Operates entirely client-side on the in-memory `BibleData` object. `initSearch()` stores a reference to the data and extracts `bookNames` ‚Äî it does not create any intermediate arrays or duplicate verse text, keeping memory usage minimal.

**Query parsing:** Semicolon-separated terms (after normalization). Each term can be:
- A book reference (with optional chapter, chapter range, verse, verse segments)
- A quoted text filter (`"grace"`)
- A combined reference + text filter (`Romans "faith"`)
- A Strong's number (`G2316` or `H430`) ‚Äî searches interlinear data for matching words

**Query normalization (`normalizeQuery`):** Applied before splitting on semicolons in `search()`, `tryParseNav()`, and `parseQueryBooks()`:
- Replaces en-dash (`\u2013`) and em-dash (`\u2014`) with regular hyphens.
- Converts comma-separated book references to semicolons: `, ` followed by a letter or numbered-book start (e.g., `Matt. 27:1-38, Luuk. 23:39-43` ‚Üí `Matt. 27:1-38; Luuk. 23:39-43`). Verse-segment commas (`, 5`, `, 10-15`) are left intact because they are followed by bare digits.

**Book matching pipeline (`matchBook`):**
1. Strip abbreviation periods from input (e.g., `Matt.` ‚Üí `Matt`, `Joh.` ‚Üí `Joh`). Periods after digits like `1.` are preserved.
2. Exact full-name match on English keys (longest-first to prefer "1 John" over "John").
3. Exact/starts-with match on translation aliases (Finnish display names, abbreviations).
4. Normalized alias match ‚Äî aliases with periods are compared after stripping periods (e.g., `ap. t.` ‚Üí `ap t`).
5. Prefix/abbreviation match on English keys and aliases.
6. Fuzzy match via Levenshtein distance (threshold: ‚â§1 edit for names ‚â§5 chars, ‚â§2 edits otherwise, minimum 3 chars).

**Reference parsing (`parseRef`):**
- Strips trailing incomplete operators (`-`, `,`, `:`) for progressive typing support.
- Supports: book only, book + chapter, book + chapter range, book + chapter:verse, book + chapter:verseSegments (ranges and comma-separated).

**Text matching (`buildTextMatcher`):**
- Plain substring (fast `.includes()` path) when no anchors.
- `^word` ‚Äî word-start boundary (`\b` prefix). `word$` ‚Äî word-end boundary. `^word$` ‚Äî exact word.
- Special regex characters are escaped.

**Results:** Returns `VerseResult[]` with deduplication via a `Set<string>` of `book:chapter:verse` keys. No limit on result count ‚Äî all matches are returned.

**`tryParseNav(query)`:** Attempts to parse the entire query as pure references (no text filter). Returns `NavRef[]` if successful, `null` if any term has a quoted filter. Used by `applyState()` to decide whether to navigate or search.

**`parseQueryBooks(query)`:** Parses each term to extract the English book key, rest (chapter/verse), and quoted filter. Used during translation switching to translate book names.

## Shared Module (shared/bible-loader.ts)

Exports three items used by both `server.ts` and `build-static.ts`:

## Server (server.ts)

Bun HTTP server on port 3000. Uses `loadBible` and `discoverTranslations` from the shared module. On startup:
1. Builds the client bundle (`app.ts` ‚Üí `public/bundle.js`).
2. Loads all translations into memory from `public/text/` directory.
3. Serves:
   - `/text/bible-CODE.json` ‚Äî combined Bible data for a translation (from memory cache).
   - `/data/descriptions-LANG.json` ‚Äî book and chapter descriptions by language (from file cache).
   - `/text/translations.json` ‚Äî array of available translation codes.
   - `/text/interlinear/{Book}.json` ‚Äî per-book interlinear data (served from file).
   - `/text/strongs.json` ‚Äî Strong's Concordance dictionary (served from file).
   - Static files from `public/` with MIME type mapping.
   - SPA fallback: unknown paths serve `index.html`.
4. Path traversal protection: resolved paths must start with `PUBLIC`.

## Features in Detail

### 1. Search

The search input in the sticky header accepts multiple query formats. Input is debounced at 150ms. If the query parses as pure references (no quoted text), the app navigates directly to those passages instead of showing search results.

**Query formats:**
- `John 3:16` ‚Üí navigates to single verse
- `Genesis 1-3` ‚Üí navigates to chapter range
- `Genesis 1:1-3,5,8-10` ‚Üí navigates to specific verse segments
- `John 3:16; Rev 1:1` ‚Üí multi-reference navigation with sections and dividers
- `"grace"` ‚Üí full-text search across all verses
- `"^grace"` ‚Üí word-start boundary match
- `"grace$"` ‚Üí word-end boundary match
- `"^grace$"` ‚Üí exact word match
- `Romans "faith"` ‚Üí scoped text search within Romans
- `Gen 1-3 "light"` ‚Üí scoped text search within chapter range
- `G2316` ‚Üí Strong's number search (finds all verses with Greek word Œ∏ŒµœåœÇ)
- `H430` ‚Üí Strong's number search (finds all verses with Hebrew word ◊ê÷±◊ú÷π◊î÷¥◊ô◊ù)

**Non-standard syntax support:** The search input accepts common non-standard conventions:
- En-dashes and em-dashes (`‚Äì`, `‚Äî`) as verse/chapter range separators (normalized to hyphens)
- Periods after book abbreviations (`Matt.`, `Joh.`, `Luuk.`, `Gen.`, `Rev.`)
- Comma-separated multi-book references (`Matt. 27:1‚Äì38, Luuk. 23:39‚Äì43, Joh. 19:31‚Äì37`)
- These are normalized before parsing via `normalizeQuery()` and period stripping in `matchBook()`.

**Auto-closing quotes:** Typing `"` inserts `""` and places cursor between them. Typing `"` when cursor is before an existing `"` skips over it.

**Fuzzy matching:** Typos in book names are corrected via Levenshtein distance. `"genisis"` ‚Üí Genesis, `"revelaton"` ‚Üí Revelation.

**Search results UI:** Shows total count, then pages of 50 results. Each result shows the reference and verse text with matching terms highlighted using `<mark>` tags. Clicking a result navigates to that verse. "Show more (N)" button loads the next page.

### 2. Book Index Panel

Three-column browser opened via the magnifier icon (inside the search input) or Ctrl+I:
- **Column 1 (Books):** All books with "Old Testament", "Deuterocanonical", and "New Testament" section labels. Hovering or keyboard-navigating to a book reveals its chapters.
- **Column 2 (Chapters):** Chapter numbers with verse 1 text preview. Hovering shows verse list. Clicking navigates to that chapter.
- **Column 3 (Verses):** Verse numbers with truncated text preview (50 chars). Clicking navigates to that verse.

Keyboard navigation: Arrow up/down within columns, Arrow left/right or Tab between columns, Enter to select. The panel has scroll locking (body overflow hidden + scrollbar compensation padding).

### 3. Parallel Translation

When a secondary translation is selected in settings, content renders in a two-column CSS Grid layout. Each column shows the same passage in a different translation with its own translation label and copy button. A "Copy both" button above copies both translations with labels.

Supported parallel views: chapter, single verse, verse segments, multi-reference. Chapter ranges and full book views do not have parallel mode ‚Äî they render single-column.

In parallel mode, each column displays subheadings in its own language. The secondary translation's subheadings are loaded separately via `setSecondarySubheadings()` and selected by `renderStyledVerses()` based on the `secondary` parameter.

### 4. Verse Highlighting

Clicking a verse's superscript number opens a floating context menu with:
- A "Copy verse" button (clipboard icon + translation label + reference + text).
- Five color dots (yellow, green, blue, pink, orange). The active color has a ring indicator.
- A remove button (‚úï) appears only when a highlight exists.

Highlights persist in IndexedDB and are loaded into memory on startup. The highlight map is passed to render functions which add `hl-{color}` CSS classes to verse spans.

Long-press (500ms) on touch devices triggers the same menu.

### 5. Book & Chapter Descriptions

Each translation can include a `descriptions.json` file with summaries for books and chapters. When viewing a full chapter (single or parallel), chapter range, or entire book, a short italic description is displayed below the title and above the verses. When viewing chapter 1 (or the first chapter in a range), the book-level description is also shown above the chapter description. Descriptions are not shown for single verse views, verse segments, search results, or multi-reference navigation.

In the book index panel, the chapter column shows chapter descriptions (when available) instead of the first verse preview text.

Descriptions are sourced from the Catholic Public Domain Version (CPDV), attributed in the footer.

Descriptions are loaded via `fetchDescriptions(code)` in `app.ts` (which maps translation code to language via `TRANSLATION_LANG`) and stored in render.ts via `setDescriptions()`. They are re-fetched when switching translations. If no descriptions file exists for a translation, an empty array is used gracefully.

The description paragraph uses the `.description` CSS class: italic, sans-serif, muted color, centered.

### 6. Copy to Clipboard

**Section copy (heading button):** Copies all displayed verses with a header line showing translation label, book name, and chapter/verse reference. In parallel mode, three buttons appear: copy primary only, copy secondary only, copy both (separated by blank line).

**Verse copy (context menu):** Copies a single verse with translation label and reference prefix.

Both actions show a toast notification ("Copied!" / "Kopioitu!").

### 7. Chapter Navigation

Arrow links at top and bottom of chapter/verse views. `getChapterNav()` and `getVerseNav()` compute previous/next targets by walking the ordered book/chapter/verse structure, crossing book boundaries. Disabled arrows are greyed out spans (no link).

**Swipe navigation:** On touch devices, horizontal swipes (>80px, <500ms, more horizontal than vertical) trigger the corresponding nav arrow click.

### 8. Themes

Three options via `data-theme` attribute on `<html>`:
- **Light** ‚Äî White/off-white background, dark text.
- **Dark** ‚Äî Dark grey background, light text. Highlight colors use 0.35 opacity (vs 0.4 in light mode) for visible highlighting on dark backgrounds.
- **System** ‚Äî Follows `prefers-color-scheme` media query, updating on OS changes.

### 9. Font Size

Five levels via `data-font-size` attribute: small (14px), medium (17px), large (20px), xl (23px), xxl (26px). Affects `.verses` font-size. Saved in `localStorage` as `bible-font-size`.

### 10. Shareable URLs

Every navigation state is encoded in URL query parameters. Examples:
- `?t=NHEB&book=jhn&chapter=3&verse=16` ‚Üí John 3:16 in NHEB
- `?t=KR38&q="armo"` ‚Üí search for "armo" in KR38
- `?t=NHEB&p=KR38&book=gen&chapter=1` ‚Üí Genesis 1 with KR38 parallel
- `?t=KJV&book=jhn&chapter=1&il=1` ‚Üí John 1 in KJV with interlinear enabled

Book names use 3-character codes (`bookCodes.ts`) for compact URLs. Translation and parallel translation are included in every URL.

### 11. Print View

`@media print` CSS hides all interactive UI (header, footer, overlays, nav arrows, copy buttons, verse menu, toast). Shows a `.print-translation-label` that's hidden in screen mode. Verse text renders cleanly for printing.

### 12. Progressive Web App

Installable via browser's "Install" / "Add to Home Screen". Service worker pre-caches the app shell for instant offline startup. JSON data uses network-first with cache fallback to stay current while supporting offline use.

### 13. Internationalization

The UI language can be set independently of the Bible translation. When switching Bible translations, the UI language auto-switches to match (NHEB/KJV/CPDV‚ÜíEnglish, KR38‚ÜíFinnish). The info modal content is rebuilt from i18n strings on language change.

All user-facing strings are in `i18n.ts`. The single HTML `index.html` contains English defaults as a fallback, but `updateStaticText()` in `app.ts` overwrites all text content from i18n on initialization and language changes.

### 14. Text Formatting (Styleguide)

Verse text is formatted with paragraph breaks, poetry indentation, and stanza breaks based on a `styleguide.json` file. The styleguide was generated from USFM source files of the World English Bible (WEB) translation and is applied to all translations for consistent formatting.

**Formatting types:**
- **Paragraph breaks** ‚Äî A visual break (`0.6em` gap) before verses that start a new paragraph in prose text.
- **Poetry indentation** ‚Äî Verses in poetic books (Psalms, Proverbs, Song of Songs, prophets, etc.) render as block-level elements with 3 levels of indentation (`q1`, `q2`, `q3`).
- **Stanza breaks** ‚Äî An extra gap (`0.8em`) between poetry stanzas.

The styleguide is fetched once on app startup (`fetch("./data/styleguide.json")`) and stored via `setStyleguide()` in `render.ts`. The `renderStyledVerses()` helper applies formatting to all verse-rendering functions.

The footer attributes the formatting to the WEB translation via the `footerStyleguide` i18n string.

### 15. Subheadings

Verse subheadings provide topical headings before groups of verses (e.g., "The Creation", "The Fall of Man"). Each language has its own subheadings file in `public/data/` ‚Äî `subheadings-en.json` (English) and `subheadings-fi.json` (Finnish).

Subheadings are loaded on app startup based on the current UI language and stored via `setSubheadings()` in `render.ts`. The `renderStyledVerses()` helper renders them as `<div class="subheading">` elements above the relevant verse.

**Parallel mode:** In parallel views, each column displays subheadings in its own language. The primary translation uses `setSubheadings()` and the secondary translation uses `setSecondarySubheadings()`. `renderStyledVerses()` selects the appropriate source based on its `secondary` parameter.

Subheadings are shown in chapter, book, chapter range, and verse segment views. They are not shown in single verse or search result views.

### 16. Interlinear View

When interlinear mode is toggled on (via the `.il-toggle-btn` button), verses display original Hebrew (OT) or Greek (NT) text word-by-word alongside the English translation. Each `.il-word` element shows the English word, original script, transliteration, and Strong's number. Clicking a word opens the Strong's Concordance panel (`#strongs-panel`) with detailed information.

Interlinear data is loaded per-book from `/text/interlinear/{Book}.json` and cached in IndexedDB. The `setSearchInterlinearData()` function in `search.ts` enables Strong's number searching (e.g., `G2316`) against the loaded interlinear data.

The interlinear toggle state is persisted in the URL via the `il=1` parameter.

### 17. Strong's Concordance Dictionary

A standalone `dictionary.html` page provides a browsable reference for all Strong's Hebrew and Greek word definitions. The Strong's data is loaded from `strongs.json`, which contains entries keyed by lowercase Strong's numbers (e.g., `g2316`, `h430`) with definitions, pronunciations, parts of speech, and cross-references.

Within the main app, the `#strongs-panel` displays individual word definitions when clicking interlinear words. The footer includes a link to the full dictionary page.

### 18. Share Links

A share button on verses and sections allows copying a link to the current passage. Two options are available:
- **Share with text** ‚Äî copies the URL along with the verse text
- **Share without text** ‚Äî copies just the URL

A toast notification confirms the link was copied. The share functionality uses the i18n strings `shareWith`, `shareWithout`, and `linkCopied`.

## Design Choices

1. **No framework:** Vanilla TypeScript for minimal bundle size and full control. DOM manipulation via innerHTML and template strings.
2. **Bun runtime:** Used for HTTP server, bundler, test runner, and file operations. No separate webpack/rollup/jest tooling.
3. **Client-side search:** All Bible data loaded into memory (~4MB per translation). Search operates directly on the `BibleData` object with no intermediate arrays ‚Äî instant substring matching or regex, no server round-trips.
4. **IndexedDB caching:** Bible data fetched once and persisted. Subsequent visits load from IndexedDB, making the app work fully offline.
5. **Canonical English keys:** Internal book keys are always English (e.g., "Genesis"). Translation-specific display names and aliases are layered on top via `bookNames.ts`.
6. **3-character URL codes:** URL book parameters use compact codes (`gen`, `jhn`, `rev`) to keep URLs short and language-independent.
7. **Minimal build dependencies:** `@types/bun`, `@types/node`, `@playwright/test`, `lefthook`, `oxlint`, `oxfmt`, and `typescript` as devDependencies. No runtime npm packages.
8. **Static deployment:** The `docs/` output is a self-contained static site. GitHub Pages serves it directly. The Bun server is dev-only.
9. **CSS custom properties:** All theming via CSS variables, toggled by `data-theme` attribute. No CSS-in-JS.
10. **oxlint & oxfmt:** Rust-based linter (`oxlint`) and formatter (`oxfmt`) with zero config. Run via `bun run lint` and `bun run fmt`. All source files pass with 0 warnings and 0 errors.
11. **Lefthook git hooks:** Pre-commit hook runs lint, format check, and typecheck in parallel. Pre-push hook runs unit tests. Hooks are auto-installed on `bun install` via the `postinstall` script.
12. **GitHub Actions CI:** Runs typecheck, lint, format check, unit tests, and e2e tests on push to `main` and on pull requests. Uses `oven-sh/setup-bun` for Bun installation.

## Adding a New Translation

1. Add a JSON file `public/text/NEWCODE.json` in the array-based source format (with `books` array containing `name`, `chapters`, and `verses` arrays).
2. Add book display names and aliases in `bookNames.ts` under a new `const NEWCODE: Record<string, BookEntry>` and add it to the `TRANSLATIONS` object.
3. Add translation metadata to `TRANSLATION_NAMES` and `TRANSLATION_LANG` in `app.ts`.
4. The build script auto-discovers translations from the `public/text/` directory.
5. Run `bun run build:static` to regenerate `docs/`.
